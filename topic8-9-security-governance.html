👤 IAM — Identity & Access Management
👥 IAM Core Concepts
  • Global service — not region-specific
  • Root account: created by default. Never use or share!
  • Users: real people. Can belong to multiple groups.
  • Groups: contain only users (not other groups)
  • Roles: assigned to AWS services to act on your behalf
IAM Policy Structure:
Version: "2012-10-17" // always this
Id: optional identifier
Statement: [
Sid: optional statement ID
Effect: Allow | Deny
Principal: account/user/role
Action: [list of API actions]

Condition: optional
]
Predefined SageMaker Policies:
AmazonSageMakerReadOnlyAmazonSageMakerFullAccessAdministratorAccessDataScientist
Access rule: Explicit DENY always wins over any ALLOW. IAM principal accesses S3 if: (IAM policy ALLOWS OR resource policy ALLOWS) AND no explicit DENY.
🔐 Password Policy & MFA
Password Policy Options:
  • Minimum password length
  • Require: uppercase, lowercase, numbers, non-alphanumeric
  • Allow users to change their own passwords
  • Require password expiration after N days
  • Prevent password re-use
MFA Devices:
Virtual MFAGoogle Authenticator (phone only), Authy (phone only). Multiple tokens per device.
U2F KeyYubiKey (Yubico). Multiple users per key.
Hardware Key FobGemalto (3rd party). GovCloud: SurePassID.
SageMaker IAM Permissions:
CreateTrainingJobCreateModelCreateEndpointConfigCreateTransformJobCreateHyperParameterTuningJobCreateNotebookInstanceUpdateNotebookInstance
Principle of Least Privilege: Grant only what's needed. Start broad in dev, lock down in prod. Use IAM Access Analyzer to generate least-privilege policies from actual access activity.

Data Masking & Anonymization:
  • Masking: obfuscate data (last 4 digits of CC#). Supported in Glue DataBrew + Redshift.
  • Anonymization techniques: replace with random, shuffle, deterministic/probabilistic encrypt, hashing
SageMaker Data at Rest:
  • KMS keys accepted by: notebooks, training/tuning/batch/endpoints
  • Everything under /opt/ml/ and /tmp can be encrypted with KMS
SageMaker Data in Transit:
  • All traffic supports TLS/SSL
  • Inter-node/inter-container traffic encryption = optional. May increase training time + cost.

Key Policies:
  • Without a key policy: you CANNOT control access to the key at all
  • Default policy: root user = full access to key (entire account)
  • Custom policy: define users/roles, admin rights, cross-account access
Cross-region snapshot copy: Snapshot encrypted with KMS Key A in eu-west-2 → copy to ap-southeast-2 → ReEncrypt with KMS Key B (different region key). Cross-account: attach KMS Key Policy to authorize + share snapshot.
🛡 SageMaker VPC & Security
SageMaker + VPC:
  • Training jobs run in a VPC by default
  • Use private VPC for extra security → need S3 VPC endpoints
  • Notebooks are Internet-enabled by default ⚠️ security hole!
  • Disable internet → need PrivateLink interface endpoint OR NAT Gateway for training/hosting
  • Training/Inference containers = Internet-enabled by default
  • Network isolation = option BUT prevents S3 access
AWS Macie:
  • ML-powered PII discovery + protection in S3
  • Scans S3 → identifies sensitive data (PII) → alerts via EventBridge
  • Fully managed, uses ML + pattern matching
AWS Secrets Manager:
  • Store secrets securely (not in code!)
  • Force rotation every X days → auto-generates new secrets (via Lambda)
  • Secrets encrypted with KMS
  • Native integration: RDS (MySQL, PostgreSQL, Aurora)
  • Multi-region: replicate secrets across regions, promote replica to standalone
SageMaker Logging: CloudWatch = invocations, endpoint latency, node health. CloudTrail = API calls + user actions in SageMaker → delivered to S3.
🌐 VPC, WAF, Shield & Network Security
🌐 VPC Networking
🏘VPCRegional. Private network for your resources.
📦SubnetsAZ-level. Public = internet accessible. Private = not.
🚪Internet GatewayVPC-level. Enables public internet access.
🔄NAT GatewayAWS-managed. Private subnet → internet (one-way). Stays private.
🔥NACLStateless. Subnet-level. ALLOW + DENY. IP addresses only.
🛡Security GroupStateful. Instance/ENI level. ALLOW only. IPs + security groups.
📡VPC Flow LogsIP traffic logs → S3, CloudWatch, Firehose.
🤝VPC PeeringConnect 2 VPCs privately. No overlapping CIDR. NOT transitive!
🔌VPC EndpointsPrivate access to AWS services. Gateway (S3, DynamoDB) or Interface (others).
🔗PrivateLinkExpose service to 1000s of VPCs securely. NLB + ENI. No peering/IGW/NAT needed.
🏢Site-to-Site VPNOn-prem → AWS over public internet. Encrypted. Fast setup.
⚡Direct ConnectOn-prem → AWS via private physical connection. Fast + secure. Takes ≥1 month.
NACL vs Security Group: NACL = stateless (check both directions), subnet level, ALLOW+DENY. SG = stateful (return traffic auto-allowed), instance level, ALLOW only.
🛡 WAF & Shield
AWS WAF (Web Application Firewall):
  • Layer 7 (HTTP) protection
  • Deploy on: ALB, API Gateway, CloudFront, AppSync GraphQL, Cognito User Pool
  • Does NOT support NLB (Layer 4)
  • Define Web ACL (Web Access Control List) rules
WAF Rule Types:
IP Set (10K IPs)HTTP headers/body/URISQL injectionXSSSize constraintsGeo-match (block countries)Rate-based (DDoS)
Web ACL = Regional EXCEPT for CloudFront (global). Rule Groups = reusable sets of rules.
AWS Shield:
Shield StandardFREE. Auto-enabled for all. Layer 3/4 protection (SYN/UDP floods, reflection attacks).
Shield Advanced$3,000/mo/org. Layer 3/4/7. EC2, ELB, CloudFront, Global Accelerator, Route 53. 24/7 DRP team. Auto-deploys WAF rules. Protects against cost spikes.
Fixed IP + WAF: WAF can't attach to NLB. Use Global Accelerator (fixed IP) → ALB with WAF. WebACL must be in same region as ALB.
✅ General AWS Security Best Practices
  • Use IAM: users/groups/roles with least privilege
  • Enable MFA: especially for root account
  • Use SSL/TLS for all connections
  • Use CloudTrail: log all API + user activity
  • Encrypt everything: at rest + in transit
  • Be careful with PII: mask, anonymize, or delete
  • Never store secrets in plaintext in code!

📊 Management & Governance — CloudWatch, CloudTrail, Config, X-Ray
📈 CloudWatch
Metrics:
  • Every AWS service. Variable (CPUUtilization, NetworkIn…)
  • Metrics → namespaces. Dimensions (instance ID, env). Timestamps.
  • Custom metrics (e.g., RAM)
Logs:
  • Log groups (app) → Log streams (instances/containers)
  • Expiration: 1 day to 10 years or never
  • Encrypted by default. KMS custom encryption optional.
  • S3 export: takes up to 12 hours (not real-time)
  • Subscriptions: real-time to KDS, KDF, Lambda
  • Logs Insights: query engine, not real-time
Alarms:
  • States: OK, INSUFFICIENT_DATA, ALARM
  • Actions: Stop/Terminate/Reboot EC2, Auto Scaling, SNS
  • Composite Alarms: AND/OR of multiple alarms (reduce noise)
SageMaker via CW: endpoint invocations, latency, node health (CPU/memory), Ground Truth worker activity.
🔍 CloudTrail
  • Governance, compliance, audit for AWS account
  • Enabled by default
  • History of events/API calls from: Console, SDK, CLI, AWS services
  • Logs → CloudWatch Logs or S3
  • Trail can apply to All Regions (default) or single region
Event Types:
Management EventsDefault logged. Operations on resources (IAM, EC2, CloudTrail). Read + Write events.
Data EventsNOT logged by default (high volume). S3 object ops, Lambda invocations.
Insights EventsDetect unusual activity. Analyzes write events vs baseline. Alerts to S3 + EventBridge.
Events retained 90 days. For longer: log to S3 + query with Athena.
Rule: Resource deleted in AWS? Investigate CloudTrail first! KMS key usage audited via CloudTrail.
⚙️ AWS Config & X-Ray
AWS Config:
  • Audit + record configuration compliance over time
  • Per-region service. Can aggregate cross-region/account.
  • Results stored in S3. Query with Athena.
  • Config Rules: AWS managed (75+) or custom Lambda
  • Rules evaluate on: config change OR schedule
  • Does NOT prevent actions (no deny) — only alerts
  • Pricing: $0.003/config item, $0.001/rule evaluation
Config Remediations + Notifications:
  • Auto-remediation via SSM Automation Documents
  • SNS notifications on non-compliance
  • EventBridge triggers on non-compliant resources
AWS X-Ray:
  • Distributed tracing for microservices debugging
  • End-to-end "request" tracking. Segments + sub-segments.
  • Visual service map. Identify bottlenecks.
  • Works with: Lambda, ECS, ELB, API Gateway, EC2
  • Import X-Ray SDK in code + install X-Ray daemon
CW vs CT vs Config: CW = performance metrics + alerts. CT = API audit trail. Config = configuration compliance over time.

AWS Budgets:

⚠️ Exam Quick-Fire — Security & Governance
🎯 Must-Know Facts
  • IAM is a global service — not region-specific
  • Never use or share the root account
  • Explicit DENY always wins over ALLOW in IAM policies
  • KMS Symmetric = AES-256. You never see the key unencrypted.
  • AWS Managed Key auto-rotates every 1 year
  • Customer Managed Key rotation must be enabled (not auto)
  • Imported KMS Key = manual rotation only via alias
  • KMS cross-region copy = ReEncrypt with destination region key
  • Without KMS Key Policy = cannot control access to the key
  • NACL = stateless (check both directions). SG = stateful.
  • VPC Peering = NOT transitive (must create separate peerings)
  • VPC Endpoint Gateway = S3 + DynamoDB only
  • PrivateLink = NLB + ENI. No peering/IGW/NAT needed.
  • Shield Standard = FREE, auto-enabled. Advanced = $3,000/mo.
  • WAF does NOT support NLB (Layer 4). ALB/API GW/CloudFront only.
  • CloudTrail enabled by default. Retained 90 days. Long-term → S3+Athena.
  • Config does NOT prevent actions — only records and alerts
  • Macie = PII discovery in S3 via ML + pattern matching
  • Secrets Manager = KMS-encrypted secrets + auto-rotation via Lambda
⚠️ Common Exam Traps
  • SageMaker notebooks are internet-enabled by default — security risk!
  • Network isolation in SageMaker prevents S3 access too
  • Inter-container traffic encryption is optional in SageMaker (may slow training)
  • KMS API limits apply with SSE-KMS (GenerateDataKey + Decrypt calls)
  • WAF Web ACL = Regional EXCEPT for CloudFront (global)
  • Shield Advanced protects against cost spikes due to DDoS — Standard doesn't
  • CloudTrail data events (S3 object ops, Lambda invocations) = NOT logged by default
  • Config Rules don't prevent actions — need AWS Config Remediation + SSM for auto-fix
  • CW Logs S3 export = up to 12 hours delay — not real-time. Use Subscriptions instead.
  • VPC Peering requires no overlapping CIDR ranges
  • Direct Connect takes at least 1 month to establish
  • Secrets Manager ≠ SSM Parameter Store (Secrets Manager = KMS + auto-rotation + RDS native)
  • IAM Access Analyzer = generates least-privilege policies from actual access patterns
  • MFA is recommended for ALL accounts especially root — not just root
