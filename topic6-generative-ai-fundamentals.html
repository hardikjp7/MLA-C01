🔄 The Transformer Architecture — Evolution & Self-Attention
📈 Evolution to Transformers
RNN / LSTM
→
Attention (RNN)
→
Transformer
Why RNNs weren't enough:
  • Sequential nature — can't parallelize
  • Information bottleneck (encoder→decoder via 1 vector)
  • Early tokens "forgotten" in long sequences
Attention with RNNs (improvement):
  • Hidden state per token (not just final state)
  • Better word order handling
  • Still sequential — still can't parallelize
Transformers (the breakthrough):
  • Ditch RNNs — use Feed-Forward Neural Networks (FFNNs)
  • Use self-attention — processes entire input at once
  • Fully parallelizable → train on vastly more data
Paper: "Attention is All You Need" (2017) — the seminal transformer paper. Killed the need for RNNs in sequence modeling.
🎯 Self-Attention Step by Step
Step 1
Each token gets embedding vector. Three weight matrices learned via backprop: Wq (Query), Wk (Key), Wv (Value)
Step 2
Multiply each token embedding by Wq→q, Wk→k, Wv→v vectors.
Step 3
Scaled dot-product attention: compute score = dot(q, each k). Apply softmax to normalize scores into percentages.
Step 4
Multiply each value (v) by its softmax score. Sum all results → updated embedding (z) for each token.
Step 5
Pass updated embeddings into FFNN layer. All tokens processed in parallel.
Multi-Headed Self-Attention:
  • Q, K, V vectors reshaped into matrices
  • Each row = one "head" → processed in parallel
  • More heads = richer contextual understanding
Masked Self-Attention (GPT):
  • Prevents tokens from "peeking" at future tokens
  • BERT uses different approach: masked language modeling (predict masked tokens)
⚡ Exam: Q = what am I looking for? K = what do I contain? V = what do I actually say? Softmax normalizes scores to attention weights.

🤖 LLM Fundamentals — How GPT Works & Key Terms
⚙️ GPT Architecture & LLM Pipeline
GPT Architecture:
  • Decoder-only — stacks of decoder blocks
  • Each block: masked self-attention layer + FFNN
  • No concept of input/output — just generates next token repeatedly
  • Trained on unlabeled text (no labels needed!)
  • Hundreds of billions of parameters
  • Prompted with tokens → keeps generating based on previous tokens
Input Processing Pipeline:
Text Input
→
Tokenization
→
Token Embedding
→
Positional Encoding
→
Decoder Blocks
  • Tokenization: Text → numerical token IDs (words or sub-words)
  • Token Embedding: IDs → vectors that capture semantic meaning
  • Positional Encoding: Adds position info using sinusoidal function (works for any length)
Output Processing:
  • Decoder outputs a vector → multiply by token embeddings
  • Get probabilities (logits) for each possible next token
  • Select next token (deterministic or randomized via temperature)
  • Repeat until end token or max length reached
⚡ Why no labels? GPT learns by predicting the next word in existing text — the "labels" are the next words in the training corpus itself. Self-supervised learning.
📚 LLM Key Terms (Exam Must-Know)
Tokens
Numerical representations of words or parts of words. LLMs think in tokens, not characters.
Embeddings
Mathematical vector representations encoding token "meaning". Similar tokens → nearby vectors in high-dimensional space.
Temperature 🌡
High = more random/creative output. Low = more deterministic/consistent output. Controls randomness in token selection.
Top P
Threshold probability for token inclusion. Higher = more random. Nucleus sampling — selects from smallest set of tokens whose cumulative probability ≥ P.
Top K
Alternate to Top P. K candidates considered for next token. Higher K = more random. Simpler than Top P.
Context Window
Max number of tokens the LLM can process at once (input + output combined). Limits how much "memory" the model has during a conversation.
Max Tokens
Hard limit for total tokens on input or output. Different from context window in some implementations.
Hallucination
When LLM generates plausible-sounding but factually incorrect information. Key challenge in production GenAI systems.
⚡ Temperature vs Top P vs Top K: All control randomness of output. Temperature = overall randomness. Top P = probability threshold. Top K = candidate count. Higher value = more random for all three.

Fine-tuning in Bedrock (Custom Models):
  • Models: Titan, Cohere, Meta (Llama)
  • Text models: prompt→completion pairs in S3
  • Image models: S3 paths to image descriptions
  • Use VPC + PrivateLink for sensitive training data
  • Can fine-tune a fine-tuned model (iterative improvement)
Continued Pre-training:
  • Like fine-tuning but with unlabeled data
  • Format: {"input": "text here"}
  • Embeds knowledge INTO the model — no need for prompt context
⚡ Fine-tuning vs RAG: Fine-tuning = bakes knowledge into model weights (expensive, permanent). RAG = retrieves at query time (cheap, updatable, can still hallucinate).

Embeddings & Vector Search:
  • Embedding = high-dimensional vector (100s-1000s of dimensions)
  • Similar items → close vectors in embedding space
  • Vector search = K-Nearest Neighbor on embeddings
  • Embedding models: Amazon Titan, Cohere
Vector DB options: Commercial: Pinecone, Weaviate. Open source: Chroma, Qdrant, Milvus, LanceDB. Existing DB with vector: OpenSearch, Elasticsearch, Redis, MongoDB, Cassandra, Neptune.

🤖 LLM Agents
🕹 Agent Architecture & Components
LLM agents = LLMs given tools to interact with the outside world. The LLM decides which tools to use and when.
Conceptual Components:
MemoryChat history + external data stores. Not persistent between sessions by default.
Planning ModuleGuidance (prompts) on how to break user requests into sub-tasks.
Tools CoreFunctions/APIs the agent can call. In Bedrock = Lambda functions.
Bedrock Agent = Action Groups + Knowledge Bases:
  • Action Groups: define tools (Lambda functions) + prompts for when to use them
  • Must define tool parameters (name, description, type, required?)
  • Description is critical — LLM uses it to extract params from user prompt
  • Agent can ask user for missing parameters interactively
  • Knowledge Bases: RAG as just another agent tool ("Agentic RAG")
  • Code Interpreter: optional — agent writes its own code for charts/computation
Deploying Bedrock Agents:
  • Create an "alias" → creates deployed snapshot
  • On-Demand Throughput (ODT): runs at account quota limits
  • Provisioned Throughput (PT): purchase higher token rate
  • Application calls InvokeAgent with alias ID
⚡ Exam: Tools = Lambda functions. Alias = deployed version. Description parameter matters for LLM tool routing. Agentic RAG = Knowledge Base as a tool. Code Interpreter = optional feature for charts/code.
⚡ Exam Quick-Fire & Traps
Critical Facts:
  • Transformer = parallelizable (RNNs are not)
  • Self-attention: Q×K = attention scores → softmax → weight V vectors → sum
  • GPT = decoder-only. BERT = encoder-only. T5 = both.
  • DistilBERT = 40% smaller via knowledge distillation
  • Positional encoding uses sinusoidal function → works on any sequence length
  • Bedrock requires model access request before use
  • Fine-tuning in Bedrock = Titan, Cohere, Meta (Llama) only
  • Continued Pre-training = unlabeled data (not prompt→completion pairs)
  • Knowledge Base default vector store = OpenSearch Serverless
  • Guardrail Contextual Grounding = measures grounding + relevance
  • Bedrock agents use Lambda for tools + alias for deployment
  • Temperature HIGH = more random. Temperature LOW = more consistent.
Common Exam Traps:
  • RAG does NOT train the model — it retrieves at query time
  • RAG can still hallucinate — it just hallucinates less
  • Bedrock agents tool descriptions are used by the LLM — description quality matters
  • Fine-tuning in Bedrock = Titan/Cohere/Meta ONLY — not Claude
  • Must be IAM user, NOT root for Bedrock
  • Watermark detection works for Titan-generated images only
  • Bedrock Studio requires SSO + IdP — not just IAM users
  • Continued Pre-training ≠ Fine-tuning. Different data formats.
  • Knowledge Base chunking controls retrieval granularity — wrong chunk size = poor retrieval
  • Provisioned Throughput = purchase higher token rate. On-Demand = account quota limits.
