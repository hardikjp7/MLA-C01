🐳 Containers — Docker, ECS, ECR, EKS
🐳 Docker & ECR
  • Apps packaged in containers → run anywhere, any OS
  • Images built from Dockerfile → stored in repos
  • ECR = AWS container registry (private + public gallery)
  • IAM controls access. Backed by S3.
  • Features: vulnerability scanning, versioning, tags, lifecycle
SageMaker Container Paths:
/opt/ml/input/config/ → hyperparameters.json
/opt/ml/input/data/<channel>/ → training data
/opt/ml/model/ → output model
/opt/ml/code/ → script files
/opt/ml/output/ → failure info
ENV vars: SAGEMAKER_PROGRAM, SM_MODEL_DIR, SM_CHANNELS, SM_HPS, SM_USER_ARGS. Install: pip install sagemaker-containers
🚢 Amazon ECS
Two Launch Types:
EC2 LaunchYou manage EC2. Each instance needs ECS Agent. You pay for EC2.
FargateServerless. No EC2 to manage. Scale by increasing tasks. Pay for CPU/RAM used.
IAM Roles:
  • EC2 Instance Profile: ECS agent, CloudWatch logs, ECR pulls, Secrets Manager (EC2 only)
  • ECS Task Role: Per-task permissions (S3, DynamoDB, etc). Defined in task definition.
Load Balancers:
ALBmost use cases
NLBhigh throughput + PrivateLink
CLBnot recommended
EFS + ECS: Mount EFS on ECS tasks (EC2 or Fargate). Fargate + EFS = fully serverless. S3 cannot be mounted as a file system!
☸️ Amazon EKS
  • Managed Kubernetes on AWS
  • Alternative to ECS — same goal, different API
  • Cloud-agnostic (also works on Azure, GCP)
  • Use when: already running Kubernetes on-prem or other cloud
  • Multi-region: one EKS cluster per region
  • CloudWatch Container Insights for logs + metrics
Node Types:
Managed GroupsAWS creates + manages nodes (EC2). ASG-backed. On-Demand or Spot.
Self-ManagedYou create + register nodes. Can use EKS Optimized AMI.
FargateNo nodes to manage at all.
Storage (CSI driver):
EBSEFS (Fargate OK)FSx for LustreFSx for NetApp
📦 AWS Batch
  • Run batch jobs as Docker images
  • Dynamic EC2 + Spot provisioning. Serverless.
  • Pay for underlying EC2 only
  • Schedule: CloudWatch Events
  • Orchestrate: Step Functions
AWS Batch vs Glue:
Glue ETLApache Spark focus. Manages resources. Data Catalog integration.
AWS BatchAny compute job (not just ETL). Must provide Docker image. Resources in your account.
Rule: ETL = Glue. Non-ETL compute jobs = AWS Batch.
🏗 Infrastructure as Code — CloudFormation, CDK, CI/CD
📋 CloudFormation & CDK
AWS CloudFormation
  • Declarative IaC — describe what you want, CF creates it in order
  • JSON/YAML templates. Supports almost all AWS resources.
  • Resources tagged with stack identifier for cost tracking
  • Destroy + recreate infra on the fly (dev cost savings!)
  • Auto-generates architecture diagrams
  • Use AWS::SageMaker::Model for repeatable SageMaker deployments
AWS CDK (Cloud Development Kit)
  • Define infra using: JS/TypeScript, Python, Java, .NET
  • High-level constructs → compiled to CloudFormation (JSON/YAML) via cdk synth
  • Deploy infra + application code together
  • Great for: Lambda, Docker containers in ECS/EKS

⚡ Amazon EventBridge & Step Functions
Amazon EventBridge
  • Schedule cron jobs OR react to AWS service events
  • Three bus types: Default (AWS), Partner, Custom
  • Event buses accessible cross-account via resource policies
  • Archive + replay events
  • Schema Registry: auto-infer event schemas, generate code
Destinations: Lambda, AWS Batch, ECS Task, SQS, SNS, Kinesis, Step Functions, CodePipeline, CodeBuild, SSM, EC2
AWS Step Functions
  • Visual workflow designer for complex pipelines
  • Advanced error handling + retry outside of code
  • Audit history of all workflow executions
  • Max execution time: 1 year
State Types:
TaskChoiceWaitParallelMap ⭐ (for ML!)PassSucceedFail
Map state = run steps for each item in dataset in parallel. Works with JSON, S3 objects, CSV. Perfect for ML batch processing!
🌊 Orchestration — MWAA (Airflow) & Lake Formation
🌀 Amazon MWAA (Managed Apache Airflow)
  • Managed Apache Airflow — no install/maintenance
  • Workflows defined as Python DAGs (Directed Acyclic Graphs)
  • DAG files uploaded to S3 → MWAA picks them up
  • Runs in a VPC (at least 2 AZs). Private or public endpoints.
  • IAM-managed access. Automatic scaling of workers.
  • Schedulers + Workers = AWS Fargate containers
Integrations:
AthenaBatchEMRFargateEKSKinesisGlueLambdaRedshiftSageMakerS3SNSSQS
Use Cases:
Complex ML workflowsETL coordinationML data preparation
MWAA vs Step Functions: MWAA = Python-defined DAGs for complex batch workflows. Step Functions = visual state machines for event-driven/real-time workflows. Both can orchestrate ML pipelines.
🏞 AWS Lake Formation
  • "Set up a secure data lake in days"
  • Built on top of Glue — everything Glue can do + more
  • Data loading, monitoring flows, partition management
  • Encryption, key management, access control, auditing
  • Lake Formation itself is FREE — pay for underlying services (Glue, S3, EMR, Athena, Redshift)
Fine-Grained Security:
Row-levelAll columns + row filter. Only specific rows visible.
Column-levelAll rows + specific columns. Only certain columns visible.
Cell-levelSpecific rows + specific columns. Most granular.
Policy tagsTag-based permissions on databases, tables, columns.
Governed Tables (ACID):
  • Supports ACID transactions across multiple tables (new S3 table type)
  • Works with streaming data (Kinesis). Query with Athena.
  • Row and Cell-Level Security for governed + S3 tables
  • Automatic compaction for storage optimization
⚡ Cross-account: Recipient must be data lake admin. Use RAM for external org accounts. IAM permissions needed for KMS keys on encrypted catalogs.
🚀 SageMaker Deployment — Inference Options & Monitoring
📡 Inference Types
TypeUse WhenKey Facts
Real-timeInteractive, low latency requirementsPersistent endpoint. Individual predictions on demand.
ServerlessIdle periods, unpredictable traffic, tolerates cold startsAuto-provision, scale to zero. Monitor: ModelSetupTime, Invocations, MemoryUtilization.
AsyncLarge payloads, long processing (near-real-time OK)Queued requests. Payload up to 1 GB. Queued processing.
Batch TransformPredictions for entire dataset at onceNot an endpoint. Process large datasets offline.
Deployment Methods:
  • JumpStart: Pre-trained models to pre-configured endpoints
  • ModelBuilder (Python SDK): Configure deployment from code
  • CloudFormation: AWS::SageMaker::Model for repeatable deployments
HA: SageMaker auto-distributes across AZs — but needs >1 instance. Deploy multiple instances per endpoint. VPC with 2+ subnets in different AZs.
🔵 Deployment Safeguards
Production Variants (A/B Testing):
  • Test multiple models on live traffic using variant weights
  • e.g., new model at 10% → ramp to 100% when confident
  • Validates in real-world settings (offline validation isn't always useful)
Blue/Green Deployments (Deployment Guardrails):
All at Once
Shift all traffic to new model, monitor, terminate old fleet
Canary
Shift small % of traffic first. Monitor. Expand if OK.
Linear
Shift traffic in linearly spaced steps. Gradual rollout.
Shadow Tests:
  • Compare performance of shadow variant vs production
  • You monitor in console and decide when to promote
Auto-rollbacks: Deployment Guardrails includes automatic rollback on failure. Always load test your scaling configuration before production!
🛠 Inference Pipelines & Tools
Inference Pipelines:
  • 2–15 containers in a linear sequence
  • Mix of built-in algorithms + custom Docker containers
  • Pre-processing → Predictions → Post-processing
  • Spark ML + scikit-learn containers supported
  • Spark ML serialized to MLeap format
  • Supports both real-time + batch transform
SageMaker Neo (Edge Deployment):
  • Train once, run anywhere (edge devices)
  • ARM, Intel, Nvidia processors — compiles to hardware-optimized code
  • Frameworks: TF, MXNet, PyTorch, ONNX, XGBoost, DarkNet, Keras
  • Deploy to HTTPS endpoint OR IoT Greengrass for actual edge devices
  • Greengrass: inference at edge with local data, model trained in cloud. Uses Lambda inference apps.
  • Must use same instance type for compilation AND hosting
Inference Recommender:
  • Instance Recommendations: ~45 min. Load tests on recommended instances.
  • Endpoint Recommendations: ~2 hrs. Custom load test with your traffic patterns.
Automatic Scaling: Set target metrics + min/max capacity + cooldown. Works with CloudWatch. Load test BEFORE using!
🔁 MLOps Pipelines — SageMaker Projects, Pipelines & Kubeflow
🏭 SageMaker Projects & Pipelines
SageMaker Projects (native MLOps CI/CD):
  • SageMaker Studio's native MLOps solution
  • End-to-end ML lifecycle: build → prep → train → evaluate → deploy → monitor → update
  • Uses code repositories + SageMaker Pipelines defining steps
  • Supports: images, feature engineering, training, evaluation, deployment, monitoring
SageMaker Pipelines (workflow automation):
  • Define ML workflow steps as a DAG
  • Each step = a SageMaker job (processing, training, evaluation, deployment)
  • Integrates with Model Registry for approval workflows
MLOps with Kubernetes:
  • SageMaker integrates with Kubeflow Pipelines via SageMaker Components
  • Enables hybrid ML workflows (on-prem + cloud)
  • Integrate existing ML platforms built on Kubernetes/Kubeflow
MLOps Workload Orchestrator (Solutions Library):
  • CloudFormation templates for MLOps pipelines
  • 12 pipelines: training with HPO/Autopilot, BYOM inference, custom Docker, Model Monitor
⚡ SageMaker Projects workflow: Create Model Group → SageMaker Pipeline → Register model versions → Model Registry Collections → Review/Approve → Deploy via CI/CD.
⚠️ Exam Quick-Fire & Traps
Critical Facts:
  • ECS Fargate + EFS = fully serverless container + storage
  • S3 cannot be mounted as a file system to ECS tasks
  • CDK compiles to CloudFormation via cdk synth
  • CloudFormation AWS::SageMaker::Model = repeatable endpoint deployments
  • Step Functions Map state = parallel processing per dataset item (key for ML!)
  • Step Functions max execution = 1 year
  • MWAA schedulers/workers run on Fargate containers
  • Lake Formation = free (pay for Glue, S3, EMR, Athena, Redshift)
  • Governed Tables in Lake Formation = ACID + can't change type after creation
  • Neo requires same instance type for compilation AND deployment
  • Async inference = max 1 GB payload size
  • SageMaker HA = needs >1 instance + VPC with 2+ subnets in different AZs
  • Inference Pipeline = 2–15 containers. Spark ML → MLeap format.
Common Exam Traps:
  • ECS EC2 Instance Profile ≠ ECS Task Role — different purposes, different scope
  • ALB = ECS standard. NLB = high throughput/PrivateLink. CLB = deprecated.
  • AWS Batch = any Docker job. Glue = ETL specifically. Don't swap them.
  • GitHub Flow = only 2 branch types (main + feature). Git Flow = 5 branch types.
  • Lake Formation cross-account: recipient must be data lake admin AND need RAM for external orgs
  • EKS vs ECS: EKS = Kubernetes (cloud-agnostic). ECS = AWS-native. Similar goal, different API.
  • Serverless Inference = tolerates cold starts. Not suitable for consistent low-latency.
  • Blue/Green = Canary ≠ Linear ≠ All at Once — know the differences!
  • Production Variants = A/B testing on live traffic. Shadow Tests = shadow variant performance comparison.
