⚡ The 3 V's of Data
  • Volume — Amount of data (GB → Petabytes). Challenge: storing & processing at scale.
  • Velocity — Speed of data generation. High velocity = real-time/near-real-time processing needed. e.g., IoT sensors, HFT
  • Variety — Different types & sources. Mix of structured, semi, unstructured.
⚡ Exam: Know which AWS service solves which V. Kinesis = Velocity, S3 = Volume, Glue = Variety.

🪣 Amazon S3 — Core Storage
🪣 S3 Fundamentals
  • Buckets: Globally unique name, region-level resource
  • Objects: Max 5TB. Key = full path (prefix + name)
  • Multi-part upload: Recommended >100MB, required >5GB
  • Durability: 99.999999999% (11 9's) — all classes
  • No "directories" — just long keys with slashes
PUT/POST/DELETE per prefix/s3,500
GET/HEAD per prefix/s5,500
Latency100–200ms
⚡ Exam: S3 performance = spread reads across multiple prefixes. 4 prefixes × 5,500 = 22,000 GET/s.
🔐 S3 Security
  • IAM Policies: User/role-based, API-call level permissions
  • Bucket Policies: JSON, resource-based, cross-account access
  • ACLs: Object or bucket level (can be disabled)
  • Access Points: Separate DNS + policy per prefix/team
Access Rule:
IAM ALLOW OR Resource Policy ALLOW
AND NO explicit DENY → ✅ Access
⚡ Exam: Block Public Access can be set at account level. Bucket Policy evaluated before Default Encryption.
⚙️ S3 Key Features
  • Versioning: Bucket-level. Protects from deletes. "null" version before enabling. Suspend doesn't delete versions.
  • CRR (Cross-Region): Compliance, low latency, cross-account
  • SRR (Same-Region): Log aggregation, test/prod sync
  • S3 Object Lambda: Transform objects on retrieval via Lambda. Use for PII redaction, format conversion.
  • Transfer Acceleration: Routes via edge location. Compatible with multi-part.
  • Byte-Range Fetches: Parallel GETs, resilience, partial data retrieval.
🚫 Replication: Only NEW objects after enabling. No chaining (B1→B2→B3 doesn't replicate B1 to B3). Delete markers optional.

📅 S3 Lifecycle Rules & Events
  • Transition Actions: Move between storage classes after N days
  • Expiration Actions: Delete objects/versions after N days
  • Can target: Prefix (e.g., s3://bucket/mp3/*) or Tags
  • S3 Analytics: Recommends transitions for Standard→Standard-IA. Daily report. 24–48h to start. Does NOT support One-Zone-IA or Glacier.
Event Notifications:
  • Events → SNS, SQS, Lambda (need resource policies)
  • Via EventBridge → 18+ AWS services, JSON filtering, archive/replay
  • Delivery usually in seconds (sometimes >1 minute)
📋 Scenario: Recover deleted objects immediately for 30 days, then within 48h for 1 year → Enable Versioning + Standard-IA (noncurrent) + Glacier Deep Archive.
💾 Block, File & Parallel Storage
💽 EBS — Elastic Block Store
  • Network drive attached to EC2 (not physical)
  • Locked to AZ — can't move without snapshot
  • 1 instance at a time (except io1/io2 multi-attach)
  • Provisioned capacity (billed even if unused)
  • Root volume deleted on termination by default
  • Elastic Volumes: Resize/change type without detach
  • Can only INCREASE size (not decrease)
⚡ Move EBS across AZ: Snapshot → Restore in new AZ.
📁 EFS — Elastic File System
  • Managed NFS. Mount on hundreds of EC2s across AZs
  • Linux only (POSIX / NFSv4.1)
  • Expensive (3x gp2), pay-per-use
  • Auto-scales to petabyte, no capacity planning
Performance Modes:
  • General Purpose (default): Low latency
  • Max I/O: High throughput, high latency, big data
Throughput Modes:
  • Bursting: 50MiB/s per TB + bursts
  • Provisioned: Fixed regardless of storage
  • Elastic: Auto-scale (unpredictable workloads)
EFS vs EBS: EFS = shared multi-AZ NFS (Linux). EBS = single-instance block storage.

🔥 Data Firehose & Apache Flink
Amazon Data Firehose (formerly Kinesis Data Firehose)
  • Fully managed, no data storage, no replay
  • Near real-time (buffered), auto-scaling, serverless
  • Destinations: S3, Redshift, OpenSearch, Splunk, Datadog, HTTP endpoint
  • Transforms: CSV→Parquet, gzip/snappy compression
  • Custom transforms via Lambda
Managed Service for Apache Flink (MSAF)
  • Formerly "Kinesis Data Analytics for Apache Flink"
  • Write custom Flink apps (Java, Python, Scala)
  • Sources: Kinesis, MSK | Sinks: Kinesis, Firehose, S3
  • Use cases: Streaming ETL, real-time analytics, metrics
  • RANDOM_CUT_FOREST = SQL anomaly detection on streams
⚡ Kinesis Streams vs Firehose: Streams = real-time + storage + replay + producer/consumer code. Firehose = fully managed + near-real-time + no storage + no replay.

🎯 Exam Quick-Fire — Data Ingestion & Storage
🎯 Key Facts to Memorize
  • S3 durability = 11 9's (99.999999999%) — all classes
  • S3 Standard availability = 99.99%
  • Kinesis shard = 1 MB/s in, 2 MB/s out
  • S3 max object size = 5 TB; multi-part required >5 GB
  • Glacier Deep Archive = 180 day min storage duration
  • S3 Glacier Flexible Retrieval = 90 day min, Bulk is FREE
  • EFS = Linux only (POSIX), multi-AZ, more expensive than EBS
  • EBS = locked to AZ, 1 instance (except io1/io2 multi-attach)
  • FSx for Lustre integrates with S3 for ML training data
  • Parquet saves 30–90% cost in Athena queries
  • RANDOM_CUT_FOREST = anomaly detection in Kinesis Analytics
  • Data Lake = schema-on-read (ELT), Warehouse = schema-on-write (ETL)
⚠️ Common Exam Traps
  • S3 is global service in UI but buckets are region-level
  • Replication: only new objects after enabling. No chaining.
  • S3 Analytics only recommends Standard → Standard-IA, NOT Glacier
  • SSE-C requires HTTPS (mandatory, not just recommended)
  • Kinesis Firehose = near real-time (buffered), NOT real-time
  • MSK can only add partitions, cannot remove them
  • EFS supports Linux only — NOT Windows
  • One Zone-IA: data lost if AZ is destroyed
  • FSx Lustre Scratch = NOT replicated (temporary). Persistent = replicated within AZ.
  • Data Mesh = governance paradigm, not a specific technology
  • KMS quota: SSE-KMS uploads/downloads count toward KMS API limits
  • S3 Object Lambda = transform on retrieval (not on write)
